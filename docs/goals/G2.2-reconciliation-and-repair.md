# G2.2 对账、补洞与修复设计文档

> 作者：Codex ｜ 日期：2026-07-24 ｜ 状态：Proposed（等待产品合同确认）

## 1. 背景与问题（Context）

G2.1 已把 Telegram Desktop JSON 接入 Bot 共用的 source-neutral message writer，并交付不可变
revision、current projection、source observation、content fingerprint、import run 和隐私安全报告。
这解决了“历史内容如何进入同一模型”，但尚未回答“如何持续发现和修复证据之间的不一致”。

当前系统有五类需要对账的状态：

1. Bot polling cursor、durable inbox/task 与已落库 observation 之间的 transport/processing 风险；
2. 多次相同或重叠 Desktop import 与既有 observation 之间的 run lineage；
3. Bot 与 Desktop 对同一 revision 提供的不同媒体 locator/availability evidence；
4. stale/conflict observation 与 current projection 之间需要 owner 判断的差异；
5. current revision pointer、rendered HTML/version 等可确定性重建的派生状态。

Telegram Bot API 不提供频道历史遍历，`update_id` 也是一个 Bot 的全局序列而非单频道序列。当前
poller 又只订阅 `channel_post` / `edited_channel_post`，未知或 disabled 频道按 default-deny
不保存 raw。因此数字跳跃、频道 message ID 空洞、一次 Desktop export 缺失都只能是 evidence，
不能成为自动创建消息或自动删除归档的事实。历史补洞必须继续由用户提供 Desktop export，
suite 不运行 MTProto/TDLib 用户账号，也不抓取 `t.me/s` 作为权威来源。

## 2. 目标与非目标（Goals / Non-Goals）

### Goals

- 建立独立、可复用的 reconciliation module，统一供 CLI、worker 及产品合同选择的 Admin HTTP /
  Owner Desk 使用。
- 自动、幂等地扫描并分类 gap、lineage、media evidence、stale/conflict 和 derived drift。
- 默认 scan/dry-run 只读；所有 apply 都有明确权限、reason、audit、bounded batch 和锁。
- 复用 G2.1 Desktop importer/source-neutral writer 补洞，不建立第二套 message 写入路径。
- 保持 raw、revision、source observation、fingerprint 和历史 task evidence 不可变。
- 修复可以从现有事实唯一推导的 provenance/current/renderer 状态，不伪造 Telegram evidence。
- 按第 9 节确认的操作面，为 owner 提供 CLI 或简洁 GUI，展示状态、分类 finding、恢复指引和
  被授权的操作。
- 形成 G2.3 可消费的多来源媒体 evidence，而不在 G2.2 下载媒体二进制。

### Non-Goals

- 不运行 MTProto/TDLib、Telegram user session、私有频道、评论或网页 crawler。
- 不自动从 Telegram 回捞频道历史，不制造 synthetic update/message。
- 不从数字 gap 或未声明完整覆盖的一次 export 缺失自动推断删除。
- 不 hard purge，不原位改写 immutable revision/raw/observation/task history。
- 不下载、hash、缓存或生成媒体缩略图；这些属于 G2.3。
- 不实现搜索/RSS、Astro adapter 或文章发布。

## 3. 约束与既有合同（Constraints）

- 历史来源是用户显式选择频道的 Telegram Desktop JSON；Bot 只负责持续采集。
- canonical identity 是 channel ID + Telegram message ID；suite message UUID、`publishedAt` 稳定。
- revision 不可变，current projection 独立；删除采用 tombstone 而非硬删除。
- Desktop 缺失不能在 import 内推断删除；stale/conflict 不覆盖 current。
- allowlist default-deny；disable 只停止未来采集，既有归档继续可读。
- PostgreSQL 18、Drizzle generated SQL、Node 22、单 Bot/worker、PostgreSQL durable tasks 保持不变。
- raw/source path/Telegram file ID 等 evidence 只允许 owner 主动读取，响应 `private, no-store`。
- CLI 入口保持 `kodama`；默认危险操作不执行，脚本可依赖稳定 exit code 和 versioned JSON。

## 4. Evidence 分类

finding 必须表达“系统知道什么”和“系统不知道什么”，不能折叠为一个 `has_gap` 布尔值。

| Kind | 含义 | 可信度 | 自动动作上限 |
| --- | --- | --- | --- |
| `durable_pending` | inbox/task 已持久化但尚未完成 | 确定 | 复用既有 retry/worker |
| `durable_blocked` | task 达到 terminal/blocked | 确定 | owner 重试或跳过；不改历史 |
| `operator_skipped` | owner 已确认跳过，raw/task 仍存在 | 确定 | 可从原 raw 重新走 writer |
| `disabled_window` | disabled 期间未采集的风险窗口 | 风险 | 引导 Desktop export |
| `retention_risk` | 离线/失败可能超过 Bot retention | 风险 | 引导 Desktop export |
| `transport_id_discontinuity` | Bot 全局 update ID 不连续 | 弱信号 | 记录 candidate，不映射频道内容 |
| `message_id_candidate` | 频道 message ID 范围有空洞 | 弱信号 | 记录 candidate，不创建/删除 message |
| `desktop_absence_candidate` | 显式覆盖范围中未观察到既有 message | 弱信号 | 仅形成删除候选 |
| `observation_stale` | 新 evidence 时间明确早于 current | 确定差异 | 保持 current |
| `observation_conflict` | 时间相同/未知但 fingerprint 不同 | 确定差异 | 保持 current，等待 owner |
| `media_evidence_missing` | source observation 缺少其媒体 evidence | 可重建或未知 | 只在可从 immutable raw 唯一重建时修复 |
| `derived_html_drift` | HTML/version 与 current revision renderer 不一致 | 确定 | 可确定性 rerender |
| `current_pointer_invalid` | current 不指向同 message 的有效 revision | 确定 | 可按不可变 revision 序列修复 |

`update_id` discontinuity 不能按频道归因：其他 update 类型、其他频道、过滤、24 小时 retention
和 Telegram 的长期空闲 reset 都会制造合法跳跃。历史数据库又只保存最新 cursor 和 accepted tasks，
所以 G2.2 只能从上线后的新 receipts 改善证据，不能为旧窗口补造精度。

## 5. 方案设计（Confirmed Core）

### 5.1 模块边界

新增 `apps/server/src/reconciliation/`：

```text
types.ts       public internal types and evidence taxonomy
report.ts      versioned, bounded, privacy-safe reports
repository.ts  PostgreSQL scans, row locks, run/finding/action persistence
service.ts     scan/classify/apply orchestration and policy boundary
```

对外只暴露小接口：

```ts
interface ReconciliationService {
  scan(input: ScanInput): Promise<ReconciliationReport>;
  apply(input: ApplyInput): Promise<ReconciliationReport>;
}
```

CLI、worker 以及第 9 节选择交付的 HTTP/GUI handler 不直接写 reconciliation/message/import 表。
G2.1 importer、source-neutral writer 和既有 admin rerender operation 通过 service 注入复用。

### 5.2 Scan 合同

- 以一个或多个显式 allowlist channel 为范围，禁止隐式扫未知频道。
- 默认在 PostgreSQL `READ ONLY REPEATABLE READ` transaction 内运行，不创建 run、finding、audit
  或临时状态；一次 scan 的所有分类来自同一数据库 snapshot。
- persisted/scheduled scan 由明确模式创建 run/findings；同 stable evidence key 幂等 upsert
  `firstSeenAt` / `lastSeenAt`，不重复开 case。
- 每张表使用严格唯一 keyset cursor 和 500 行 batch 扫完整范围，不以固定 `LIMIT` 提前宣告 clean；
  finding 通过 visitor 流式进入 bounded report，不在内存积累未受限 candidate 数组。
- 报告仅含计数、opaque stable key、有限 ID 和 sanitized reason，不含正文/raw/绝对或相对
  source locator、Telegram file ID / file unique ID。
- 相同 evidence 保持既有 owner resolution；只有新的 fingerprint/source/version 才可 reopen。
- 旧 finding 不在本次范围内出现时不能自动解释成“已修复”，必须由对应 invariant 的验证器确认。

### 5.3 Apply 合同

所有 apply：

- 人工 apply 必须来自显式 `--apply` 或 authenticated HTTP action；若第 9 节选择 worker safe repair，
  scheduled worker 可作为独立 system initiator 调用同一 service；
- 带 initiator 和 reason；直接 finding action 还必须带 expected evidence version；
- 先取独立 advisory lock，再对受影响 message/finding `FOR UPDATE`；
- bounded transaction/savepoint，任一 action 写 append-only audit；
- 重跑幂等；失败不篡改原始 evidence 或 terminal task state。

无产品决策也可确定为安全的 repair：

1. 从不可变 observation/run 唯一恢复缺失的 provenance link；
2. 从 observation raw 唯一恢复缺失的 source media observation；
3. 用当前 renderer 确定性重建 current revision HTML/version；
4. 仅当 message 的 revision 序列能唯一证明目标时修复 invalid current pointer。

除非第 9 节最终接受的产品合同明确授权，否则禁止自动 apply：

- 根据 gap 创建 message/revision；
- 根据 export absence、gap 或 conflict 写 tombstone；
- 把 stale/conflict observation 直接切成 current；
- 覆盖 canonical media evidence；
- 删除 raw/revision/media/task/audit。

### 5.4 Desktop-assisted recovery

reconcile 不实现另一套 Desktop parser。CLI/GUI 的补洞动作生成：

1. 风险 channel/range 与安全 export 指引；
2. 对应 `kodama import telegram-desktop ...` dry-run 命令；
3. import report/run 链接；
4. import 后重新 scan 的建议。

缺失内容由用户显式运行 G2.1 importer `--apply`，统一 writer 决定 created/matched/stale/conflict。
reconcile 只把新的 observations 与旧 finding 重新比较。

### 5.5 多来源媒体 evidence

现有 `message_media` 是创建 revision 时的 canonical projection，不足以保存后续 matched source
提供的 path/file-ID/availability。新增 observation-scoped media evidence：

```text
message_source_media_observations
  id uuid primary key
  observation_id uuid not null
  source_kind telegram_bot_update|telegram_desktop_json not null
  position integer not null
  media_kind text not null
  availability available|exceeds_maximum_size|not_included|unavailable not null
  telegram_file_id text nullable
  telegram_file_unique_id text nullable
  desktop_source_path text nullable
  source_metadata jsonb not null
  created_at timestamptz not null
  unique(observation_id, position)
```

- observation 增加 `(id, source_kind)` unique key，media 以 `(observation_id, source_kind)` composite FK
  保证来源不能伪装。
- CHECK 保证 Bot 只有真实 `file_id` / `file_unique_id`、`availability=available` 且 path 为 null；
  Desktop 的 available 行必须只有已验证相对 path，placeholder 行必须没有 path/file ID。
- 每个新 source observation 都写其 media evidence，即使 fingerprint matched 已有 revision。
- public projection 继续只读 `message_media`，不返回 source locator。
- owner raw/provenance endpoint 才可 reveal，且 `private, no-store`。
- migration 只从已有 observation/revision/media 可唯一关联的行 backfill；无法确定的旧 Desktop path
  形成 finding，要求重新 import，不伪造。

### 5.6 Reconciliation lifecycle

不改写 `message_source_observations.resolution`。拟新增：

```text
reconciliation_runs
  id, mode, scope, status, report, initiator_kind, started_at, completed_at

reconciliation_findings
  id, stable_key, telegram_chat_id?, message_id?, observation_id?, kind, severity
  state(open|resolved|ignored), sanitized_details
  evidence_version, first_seen_at, last_seen_at, resolved_at?

reconciliation_actions
  id, run_id?, finding_id, action_kind, initiator_kind, initiator_id?
  reason, before_state, after_state, created_at
```

channel-scoped finding 直接引用 allowlist Telegram bigint ID，因此即使频道尚未产生 archive row 也能
表达 risk；Bot-global `transport_id_discontinuity` / `retention_risk` 的 channel 必须为 null，
其余 kind 必须非 null。finding details 只保存分类所需的有限 ID/range/version/count，不保存正文或
source locator。
owner resolution、自动 invariant resolution 和 ignored 都用 append-only action 表记录原因。

### 5.7 CLI

拟定命令：

```text
kodama reconcile telegram \
  --channel -1001234567890 \
  [--channel -1009876543210] \
  [--apply] \
  [--reason <text>] \
  [--json]
```

- `--channel` 可重复，必须存在于 allowlist。
- 默认临时 dry-run，不写 run/finding/audit。
- `--apply` 必须同时提供非空 `--reason`；准确 mutation 集合等待第 9 节产品合同确认。
- batch safe apply 在持有 global reconciliation lock 后重新 scan，并只作用于锁内仍成立的
  deterministic evidence，不接受过期 client snapshot；单 finding 的 owner action使用
  `findingId + expectedEvidenceVersion` optimistic check。
- human/JSON 使用相同 `schemaVersion` report。
- exit 0：clean/repaired；exit 2：有 open finding 或 item error；exit 1：fatal/interrupted。
- local CLI initiator 必须记为独立 `local_operator`，不能伪装 owner/service token。

### 5.8 Admin HTTP 与 Owner Desk（等待操作面确认）

若第 9 节选择初版 GUI，则交付的只读面：

- reconciliation summary、最后 persisted scan；
- cursor-paginated findings；
- evidence kind/severity/state、channel/message suite ID、sanitized guidance；
- owner-only observation raw reveal 复用现有 provenance 隐私合同。

GUI 至少显示 last scan/status、分类计数、finding 列表和 Desktop export/import 指引。
若选择 CLI-only，这些查询仍由 reconciliation service 保持可复用，但 G2.2 不承诺 HTTP/GUI route。
写操作、service-token scope 和 tombstone UI 同样等待第 9 节确认。

### 5.9 Scheduled scan

worker 使用 PostgreSQL durable schedule/lease 与 global reconciliation advisory lock，不能与任何
其他 reconciliation scan/apply 重叠。周期可配置，默认一小时。claim/renew/finish 以 PostgreSQL
`clock_timestamp()` 判断 due/expiry，不信任调用方时钟；每次 claim 生成不可复用 UUID lease token，
并在同一 transaction 创建 token-bound `scheduled_scan` run、写入 `claimed_run_id`。owner、token、
run 三者共同 fencing，过期 takeover 会先把旧 token-bound run 原子标记为 interrupted。
scheduled scan 只使用数据库 evidence，
不访问 Telegram history、Desktop 文件或网页，也不持有 user session。

是否自动执行 safe repair 等待第 9 节确认；在未确认前只持久化 findings/report。
若选择自动 safe repair，worker 使用固定 audit reason `scheduled_safe_repair`、独立 system initiator，
并经过与人工 apply 完全相同的 global lock、锁内 re-scan、bounded transaction 和 action audit。

## 6. 数据迁移与兼容

迁移只做 additive change，使用 `drizzle-kit generate` 提交 SQL、snapshot 和 journal。

共同需要：

- `reconciliation_runs/findings/actions`；
- `message_source_media_observations`；
- `telegram_poll_receipts`：仅为非空 Bot batch 持久化 bot ID、requested/checkpoint offset、
  returned first/last update ID、returned/accepted/ignored counts 和完成时间；与 cursor/task 在同一
  checkpoint transaction 写入，不保存 ignored raw；
- `reconciliation_schedule`：单一 Telegram schedule 的 enabled、interval、next run、
  lease owner/expiry/token、claimed run 和最后 run/status；四个 active lease 字段必须同空或同
  非空，worker 通过 token/run-bound 原子 claim 恢复定时工作；
- finding/channel/state/kind、observation media 等必要索引；
- doctor schema object/constraint 检查。

重复 import lineage 的精确 join 或 compact coverage 取决于第 9 节确认。旧数据只能做确定性 backfill：

- 已有 observation 的非空 `import_run_id` 可恢复首次 run link；
- Bot 或 Desktop revision/media 只有在 observation 的 revision、source kind 与 canonical media
  source kind 完全匹配时才恢复 media evidence；Bot locator 与 Desktop 相对 path 都必须通过新表
  的 typed CHECK，跨来源 matched observation 不从 canonical media 反推 locator；
- 无法证明的旧 matched Desktop media 不从 canonical media 反推 path；
- 旧时期 transport gap 精度标记 unknown，不制造 poll receipts。

旧应用回滚时必须仍可写 G2.1 rows；新列有兼容 default/nullability，新服务的修复器可补齐回滚期间
可确定恢复的 evidence。回滚不删除新表或已生成 findings。

## 7. 安全、隐私与并发

- allowlist default-deny；scan scope 显式；disabled archive 仍可检查。
- public API 不泄露 raw、finding details、source locator 或删除原因。
- Admin summary/findings 不含 message body、绝对 path、Telegram file ID。
- owner-only raw/reveal 主动加载并设置 `Cache-Control: private, no-store`。
- local CLI、owner session、service token、worker 使用不同 initiator kind。
- 所有 reconciliation scan/apply 使用同一个 global advisory session lock；首版有意串行，避免
  `[A,B]` 与 `[B]` 这种重叠 scope 逃逸。
- apply 在 global lock 后按 canonical channel ID、Telegram message ID、finding ID 的固定升序获取
  message/finding row lock；直接 finding action再校验 optimistic evidence version。
- import 不取 reconciliation lock；两者只在 canonical message row 交汇，并使用相同
  channel/message 升序。reconciliation 只依据锁内重新读取的事实写 action。
- observation/fingerprint/sourceKey 不原位升级；新算法写新 evidence version。

## 8. 测试策略（Testing）

### Unit

- 每种 evidence kind 的分类与 severity；
- update/message ID discontinuity 永不成为删除/创建动作；
- stable finding key、reopen/resolution state；
- bounded issue samples 与报告隐私；
- dry-run/apply exit 0/1/2；
- initiator/reason/optimistic version validation。
- poll receipt classification、空 batch 不放大写入、schedule lease claim/expiry。

### PostgreSQL 18 integration

- migration/backfill/idempotency/rollback compatibility；
- overlapping import replay lineage；
- Bot-first/Desktop-match 与 Desktop-first/Bot-match 双来源 media evidence；
- fingerprint v0/v1 不混比；
- conflict/stale observation 保持 immutable；
- derived HTML/current pointer repair 幂等；
- numeric gap / 未声明完整覆盖的 absence candidate 不创建、删除或 tombstone；
- poll receipt/checkpoint 同事务、schedule 重启续跑与 global lock 串行；
- live/import/reconcile 并发；
- finding dedupe/reopen/action audit；
- public projection与Admin report无敏感泄漏；
- scope、cursor、owner raw `private, no-store`。

### CLI / GUI / gates

- CLI dry-run/apply/replay/interrupt smoke；
- 若选择 GUI：Owner Desk summary/findings/指引与已授权操作；
- 若选择 GUI：Storybook interaction/accessibility；
- lint、typecheck、unit、PG18 integration、build；
- `pnpm db:generate` no drift；
- Compose validation/smoke。

## 9. 产品合同决策门（Open Questions）

以下选择会改变公开可见性、自动 mutation、权限或长期审计模型，未确认前本文保持 Proposed：

1. **自动化边界**
   - A（推荐）：scheduled worker 只 scan/findings；CLI/GUI 显式 apply 才执行确定性安全修复。
   - B：worker 自动执行确定性安全修复。
2. **tombstone 触发权限**
   - A（推荐）：即使声明 export 完整覆盖，absence 也只生成候选；owner 带 reason 手工 hide/unhide。
   - B：声明完整覆盖的 export absence 可自动 tombstone。
3. **tombstone 公开行为**
   - A（推荐）：列表过滤、详情 404；revision/raw/media 保留到未来显式 purge。
   - B：列表过滤、详情 410；revision/raw/media 同样保留。
4. **操作面与权限**
   - A（推荐）：初版同时交付 CLI 和 GUI；GUI 提供 findings、安全修复、owner hide/unhide；
     service token 不得 tombstone。
   - B：G2.2 CLI-only，GUI 延后。
5. **重复导入 lineage**
   - A（推荐）：`import_run_observations` 精确关联每次 run 与 observation。
   - B：只保存 compact coverage/aggregate，降低行数但失去逐条审计。
6. **conflict candidate**
   - A（推荐）：G2.2 不提升未冻结 candidate；保持 current，引导重导入或选择已有 revision。
   - B：新增冻结 normalized candidate，允许 owner promote。
