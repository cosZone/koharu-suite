# G1.5 — 交付基础运维能力

> GitHub Issue: [#10](https://github.com/cosZone/koharu-suite/issues/10) ｜ 日期：2026-07-24 ｜ 状态：Accepted

## 1. 背景与问题

G1.3 已有 singleton owner、Better Auth session 与最小 Owner Desk；G1.4 已把 Telegram update
可靠写入 PostgreSQL durable inbox，并在 poison update 连续失败 10 次后阻塞对应频道。

这套系统目前能看见 blocked 数量，却不能恢复或明确跳过；频道一旦进入 allowlist 也不能停用。
公开消息列表固定返回 50 条，没有可继续读取的 cursor；自动化只能模拟网页登录。Telegram entity
仍只有 text + structured entities，无法向 RSS、Astro adapter 或内置页面提供统一的安全 HTML。
部署故障也只能靠逐项手工排查。

G1.5 把这些 Preview 运维缺口收敛为一个纵向能力：owner 能在 GUI 处理 blocked task 和频道状态，
CLI/CI 能使用最小权限 service token，公开读取具有稳定分页与基础防护，派生 HTML 可以版本化重建，
部署者可以用 `kodama doctor` 得到不泄密的诊断。

## 2. 目标与非目标

### Goals

- singleton owner 可创建、列出和撤销 scoped service token；数据库不保存明文。
- owner session 与 Bearer token 共用一套 route permission contract。
- blocked task 必须经 owner 带理由 retry/skip，并留下可追踪审计。
- channel disable/enable 只改变未来采集，不删除公开归档。
- 从 Telegram text/entities 生成确定性、安全、可版本化 rerender 的 HTML。
- 公开消息 API 使用 opaque cursor、page-size 上限、精确 CORS allowlist 与基础限流。
- `kodama doctor` 只读验证服务启动所需的关键依赖和 schema，并保持 secret redaction。

### Non-Goals

- 自动 skip、永久 purge、reconcile、历史导入或删除检测。
- 媒体下载、搜索、RSS、Astro adapter。
- 多 owner、RBAC、OAuth、邮件找回或浏览器保存 service token。
- Redis、分布式 rate-limit service、WAF 或多 API replica 的全局强一致额度。
- 读取或保存任何外来 HTML；renderer 只接受已归档 text/entities。

## 3. 已确认合同

- GUI 使用 Better Auth session；CLI/CI 使用可撤销、数据库只存 hash 的 scoped service token。
- raw JSON 只允许 owner 或具有 `admin:read` 的 token 显式读取，响应必须 `private, no-store`。
- poison update 不自动跳过；owner retry/skip 必须有理由并记录 actor、target、action 与时间。
- skip 只解除指定频道队首，不伪装为成功 ingest；raw/error evidence 保留。
- disable 立即让 allowlist default-deny 过滤该频道的新 update；既有归档仍公开可读。
- cursor 使用 `published_at + channel_id + message_id` 的稳定倒序，并以版本化 opaque string 暴露。
- 原始 Telegram revision 不可变；rerender 只更新派生 HTML 与 `renderer_version`。
- 公开 API 默认不允许跨域；显式配置的 origin 才返回 CORS header。
- 一个 API 进程的基础匿名限流不引入 Redis；service token 另使用 token 自身 rate-limit 状态。

## 4. 候选方案与结论

| 问题 | 方案 | 优点 | 风险 / 代价 | 结论 |
| --- | --- | --- | --- | --- |
| service token | 自建 token 表与 scrypt/argon2 | 完全控制 schema | 重复实现 create/verify/revoke/expiry/rate limit | 不采用 |
| service token | Better Auth API-key plugin | hashed key、permission、expiry、revoke、rate limit 已有合同 | 增加官方 plugin package 与 migration | 采用 |
| blocked task | 直接改 task 行，不记 audit | schema 小 | 无法回答谁、为何越过 poison task | 不采用 |
| blocked task | task terminal 字段 + append-only audit | recovery 与审计分离，可验证幂等 | 多一张表 | 采用 |
| renderer | 每次请求临时生成 HTML | 无 migration | 重复 CPU，无法追踪 renderer 版本 | 不采用 |
| renderer | revision 保存派生 HTML/version | API 快，支持批量升级 | rerender 会更新派生列 | 采用 |
| cursor | offset/limit | 实现简单 | 新消息插入时重复/漏项，深分页变慢 | 不采用 |
| cursor | signed cursor | 防篡改 | 需要部署 secret，轮换会使所有 cursor 失效 | 不采用 |
| cursor | versioned base64url payload + strict validation | opaque、无额外 secret、可演进 | 客户端可修改但不能越权；必须校验 | 采用 |
| public rate limit | PostgreSQL 每请求计数 | 多实例共享 | 每个 GET 都写 DB，放大公开流量 | 不采用 |
| public rate limit | 进程内 fixed window | 无新依赖，足够保护 Preview 单 API 实例 | 重启清空，多 replica 不共享 | 采用并明确边界 |

## 5. 权限与认证

### 5.1 Service token

Better Auth 增加 API-key plugin，reference 固定为 singleton owner user ID。CLI 提供：

```text
kodama token create --name deploy --scope admin:read --expires-in 30d
kodama token list
kodama token revoke --id <api-key-id>
```

允许 scope：

| scope | permission | 用途 |
| --- | --- | --- |
| `admin:read` | `admin: ['read']` | status、blocked task、配置频道与 raw reveal |
| `ingestion:write` | `ingestion: ['write']` | retry、skip、channel enable/disable |
| `content:write` | `content: ['write']` | rerender 派生内容 |

create 至少要求一个已知 scope；name trim 后 1–64 字符。明文 key 只写 stdout 一次，list/revoke
只输出 ID、name、prefix/start、permissions、created/expiry/revoked state，不输出 hash。

Bearer header 只接受 `Authorization: Bearer <key>`。无 token 时再读取 owner session cookie；不能把
无效 Bearer 静默降级成 cookie，以免调用方以为 token 权限有效。

### 5.2 Route permission

管理 route 通过统一 guard 声明 permission：

- status、blocked task list、configured channel list、raw：`admin:read`；
- retry、skip、channel enable/disable：`ingestion:write`；
- rerender：`content:write`。

owner session 通过 singleton owner check 后拥有上述全部权限。Bearer token verify 必须同时通过
expiry、revocation、owner reference、permission 与 plugin rate limit。认证失败 `401`，身份有效但
scope 不足 `403`。

所有管理响应设置 `Cache-Control: private, no-store` 与 `Vary: Cookie, Authorization`。

## 6. Durable task recovery

### 6.1 Schema

`telegram_ingest_tasks` 增加：

| 字段 | 说明 |
| --- | --- |
| `skipped_at timestamptz null` | owner 显式 terminal skip |
| `skip_reason text null` | 1–500 字符理由 |

`operation_audit_events`：

| 字段 | 说明 |
| --- | --- |
| `id uuid primary key` | audit identity |
| `actor_type owner_session/service_token` | actor 类型 |
| `actor_id text` | owner user ID 或 API key ID |
| `action task.retry/task.skip/channel.enable/channel.disable/content.rerender` | 受控 action |
| `target_type/target_id` | task/channel/renderer |
| `reason text null` | retry/skip 必填，其余可空 |
| `details jsonb` | 仅脱敏计数与版本，不放 raw/secret |
| `created_at timestamptz` | 审计时间 |

成功 task 仍使用 `processed_at`；skip 使用 `skipped_at`，两者互斥。worker 的 runnable/head-of-line
查询把 `processed_at IS NULL AND skipped_at IS NULL` 视为 unfinished。

### 6.2 Retry

在 transaction 中锁定 task：

1. task 必须是 blocked 且未 processed/skipped；
2. 写入 `task.retry` audit；
3. `blocked_at = null`、`attempt_count = 0`、`available_at = now()`；
4. 保留 `last_error` 和 raw evidence，下一次 worker 成功后按原规则清空 task raw/error。

相同 task 在第一次 retry 后再次 retry 返回 `409 task_not_blocked`，不写第二条 audit。

### 6.3 Skip

在 transaction 中锁定 task：

1. task 必须是 blocked 且未 processed/skipped；
2. reason trim 后必须 1–500 字符；
3. 写入 `task.skip` audit；
4. 设置 `skipped_at/skip_reason`，保留 raw、attempt、blocked 和 last error evidence。

skip 后后继 task 可 claim；skip 本身不会写 `telegram_updates/messages/revisions`。同 task 重复 skip
返回 `409 task_already_resolved`。

## 7. Channel enable/disable

`telegram_channel_allowlist` 增加 `enabled boolean not null default true` 与
`disabled_at timestamptz null`。

- poller checkpoint transaction 只把 enabled channel update 写入 inbox；
- disabled update 与 allowlist 外 update 一样不保存 raw，但 Bot-wide offset 仍推进；
- public `telegram_channels/messages` 不过滤 disabled，因此既有归档保持可读；
- disable/enable 在 transaction 中锁行并写 audit；
- disable 已 disabled、enable 已 enabled 为幂等成功，不重复写 audit；
- CLI 与 Admin 调用同一 service，不能各自复制状态机。

重新启用只恢复未来 update；UI/README 必须明确停用期间的 update 可能已被 Bot offset 确认，G1.5
不会补回。

## 8. Entity renderer 与 rerender

### 8.1 输入与输出

renderer 只接受 revision 的 `text` 与 normalized entities，永不接收 HTML。JavaScript string 的
index 本身使用 UTF-16 code unit，与 Telegram entity offset/length 一致；renderer 仍必须检查：

- offset/length 是非负整数且不越界；
- entity range 不切开 surrogate pair；
- 嵌套必须完整包含，交叉 range 安全降级为 escaped text；
- 相邻 range 保持顺序。

所有普通文本先 HTML escape。标签 allowlist：

| entity | 输出 |
| --- | --- |
| bold/italic/underline/strikethrough | `strong/em/u/s` |
| spoiler | `<span class="tg-spoiler">` |
| blockquote/expandable_blockquote | `blockquote`，后者加固定 class |
| code | `code` |
| pre | `pre > code`，language 只允许安全 class 字符 |
| text_link | `a`，只允许 `http:`, `https:`, `mailto:`, `tg:` |
| url/email/phone_number | 从已显示文本生成受限链接 |
| custom_emoji/未知 entity | 不加标签，保留 escaped visible text |

链接固定增加 `rel="nofollow noopener noreferrer"`。attribute 也必须 escape。

### 8.2 Storage

`message_revisions` 增加：

- `html text null`；
- `renderer_version integer not null default 0`。

新 ingest 使用常量 `CURRENT_RENDERER_VERSION` 写入 HTML/version。迁移后旧 row 是 version 0；
`rerender` 分批选择 `renderer_version < CURRENT_RENDERER_VERSION`，从 text/entities 重建并更新
派生列。它不创建 revision、不改 `telegram_update_id`、raw 或 current revision pointer。

public DTO 在 `content` 下增加 `html`。现有 `text/entities/kind` 保持兼容。

## 9. Cursor pagination

请求：

```text
GET /api/v1/messages?channel=<suite UUID>&limit=50&cursor=<opaque>
```

- default limit 50，范围 1–100；
- response 为 `{ items, nextCursor }`；
- cursor payload：

```json
{
  "v": 1,
  "publishedAt": "2026-07-24T00:00:00.000Z",
  "channelId": "uuid",
  "messageId": "uuid"
}
```

payload 使用 canonical JSON + base64url；解析后执行 strict key、version、ISO timestamp 与 UUID
校验。cursor 的 channel 必须与 query channel 相同。SQL keyset condition：

```text
published_at < cursor.published_at
OR (published_at = cursor.published_at AND channel_id < cursor.channel_id)
OR (published_at = cursor.published_at AND channel_id = cursor.channel_id
    AND message_id < cursor.message_id)
```

查询 `limit + 1` 判断下一页；最后一项生成 next cursor。非法 cursor 统一
`400 invalid_cursor`，不得返回 decoder/SQL 细节。

## 10. Public CORS 与基础限流

配置：

- `PUBLIC_CORS_ORIGINS`：逗号分隔 canonical `http(s)` origins，默认空；
- `PUBLIC_RATE_LIMIT_MAX`：默认 120，范围 1–10,000；
- `PUBLIC_RATE_LIMIT_WINDOW_SECONDS`：默认 60，范围 1–3600；
- `TRUST_PROXY`：默认 false；只有部署者确认 Hono 只可经可信反代访问时才启用。

rate limiter 只包 `/api/v1/channels`、`/api/v1/messages*` 与公开 health API，不包 auth/admin。
key 为 production Node connection address；`TRUST_PROXY=true` 时才读取规范化的第一个
`X-Forwarded-For`。fixed-window bucket 过期后惰性清理，设置有限最大 bucket 数，避免内存无界。

超限响应：

- `429`；
- `Retry-After` 秒；
- `{ "error": { "code": "rate_limited", "message": "Too many requests" } }`。

CORS 只对精确 allowlist origin 返回 header；不允许 `*`，不允许 credentials。

## 11. `kodama doctor`

doctor 不修改任何状态，也不调用 `getUpdates`：

1. parse database/auth/Telegram/public API config；
2. PostgreSQL connect + `show server_version_num`，major 必须是 18；
3. 验证预期 migration/schema columns 存在；
4. singleton owner 存在；
5. `getMe()` 成功并与 polling state Bot ID 匹配；
6. 至少一个 enabled channel；逐个用 `getChat/getChatMember` 检查仍公开且 Bot 是管理员；
7. 输出 `[ok] / [warn] / [fail]`，关键 fail 设置 exit code 1。

输出只包含 check 名、脱敏事实与修复提示。database URL 只显示 host/port/database，不显示 user、
password/query；Bot token、auth secret、owner password、raw update 和 API key 永不输出。

单元测试用 fixture doctor dependencies；PostgreSQL integration 使用临时 PG18。验收不得对真实
频道调用 doctor。

## 12. Admin

Owner Desk 新增两个窄面板：

- **采集恢复**：blocked task 表、reason 输入、Retry/Skip 按钮、成功后刷新 status；
- **频道采集**：configured channel enabled 状态与 Enable/Disable 按钮。

状态区增加 skipped count 与 renderer stale count。消息详情优先展示安全 HTML，但 raw reveal 仍需
单独点击。增加 “Rerender outdated” 操作，展示更新数量和 renderer version。

G1.5 不在浏览器创建或显示 service token 明文；token 生命周期由 CLI 管理。

## 13. 测试策略

### Unit/API

- service-token scope parser、unknown scope、Bearer precedence、401/403；
- renderer golden：UTF-16 emoji、nested/adjacent、escape、unsafe link、code/pre、
  blockquote/spoiler、custom emoji；
- cursor round-trip、tamper、wrong channel、same timestamp boundary、limit；
- CORS exact match、rate-limit 429/Retry-After、bucket cleanup；
- doctor success/failure、redaction与不调用 `getUpdates`；
- Admin route input/409/no-store。

### PostgreSQL 18 integration

- migration rerun 与 Better Auth API-key schema；
- API key 明文不落库、verify、scope、expiry、revoke；
- retry/skip/audit atomicity、重复动作、同频道 unblock、跨频道隔离；
- disable filtering 与 archived read continuity；
- ingest HTML/version 与 rerender 只选择 stale rows；
- cursor 多页不重不漏。

### Runtime

- CLI token/channel/doctor help；
- production Admin build；
- Compose config 与 Docker build；
- fixture + PG18 operations smoke；
- 不接触新的真实 Telegram 频道。

## 14. 上线与回滚

1. 部署前备份数据库，运行 `kodama migrate`；
2. 运行 fixture/安全环境的 `kodama doctor`，生产只在部署者明确执行时检查真实 Bot/channel；
3. 启动新 server，验证公开第一页与 next cursor；
4. 用测试 blocked task 验证 retry/skip，不对真实归档制造 poison update；
5. 创建最小 scope、短 expiry token，验证后撤销；
6. disable 前明确停用期间不补消息，必要时先停止 Bot 发送来源。

代码回滚不会自动删除新增表/列。旧 G1.4 worker 不认识 `skipped_at`，可能重新看到已 skip task，
因此含 skip evidence 的数据库不能直接启动旧 worker；应先停止采集并恢复升级前备份，或人工迁移
terminal state。renderer HTML 是派生数据，可从 raw/text/entities 重建；API 回滚后多余列保持无害。
