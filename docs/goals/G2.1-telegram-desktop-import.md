# G2.1 导入 Telegram Desktop JSON 设计文档

> 作者：Codex ｜ 日期：2026-07-24 ｜ 状态：Accepted

## 1. 背景与问题（Context）

M1 已交付 Bot 实时采集、不可变 revision、公开 API、Owner Desk、基础运维和 `v0.1.0` Preview。
但 Bot API 不能遍历加入前的频道历史，未获取的 update 也只保留有限时间。已确认的历史回填路径是
Telegram Desktop JSON export；suite 不运行 MTProto/TDLib 用户会话，也不在服务器保存用户 session。

Telegram Desktop JSON 与 Bot API `Update` 不是同一种输入：

- Desktop 频道 ID 是正整数 bare peer ID，Bot API 使用 `-100…` canonical channel ID；
- Desktop 没有 Bot `update_id`、`file_id`、`file_unique_id` 或 `media_group_id`；
- Desktop `text_entities` 是富文本 segment，不是 offset/length；
- 单频道 export 根对象直接含 `messages`，全账号 export 使用 `chats.list` / `left_chats.list`；
- export 是消息最终快照，不包含完整编辑历史，且可能含 service、unsupported、`rich_message`、
  私聊、联系人和其他隐私数据。

当前 domain type 强制携带 Bot raw update/update ID，revision 也强制外键到 `telegram_updates`，
media 强制携带 Bot file ID。直接制造负数 update ID 或伪 file ID 会污染 Bot namespace，
让 G2.2 对账和 G2.3 媒体恢复失去可信来源。

## 2. 目标与非目标（Goals / Non-Goals）

### Goals

- 支持 Telegram Desktop 当前的单公开频道与全账号 `result.json` 两种根结构。
- 用户必须显式选择一个或多个已配置公开频道；绝不默认导入全部聊天。
- 命令默认 dry-run，只有 `--apply` 才写库；两者共享解析、规范化、比较和决策引擎。
- 大文件使用流式解析和有限批次事务，内存不随消息总数线性增长。
- Bot 与 Desktop adapter 汇入同一个 source-neutral snapshot/writer，同时保留真实 provenance。
- 以 canonical channel + message ID、source observation 和 content fingerprint 实现可续跑幂等。
- 新 snapshot 只在时间顺序明确时更新 current；stale/ambiguous conflict 不覆盖。
- 记录 apply run/report，输出隐私安全的人类/JSON 摘要。
- 导入期间不改变 Bot binding、cursor、inbox、heartbeat、allowlist 状态或公开 suite ID。

### Non-Goals

- 不运行 MTProto/TDLib，不导入私有频道、群组、私聊或联系人。
- 不支持从未配置到 allowlist 的 archive-only channel。
- 不读取、复制、下载或 hash 媒体二进制；只记录安全的来源元数据。
- 不猜相册分组，不从 export 缺失推断删除，不执行 gap repair/reconcile。
- 不承诺恢复 Desktop export 之前的编辑 revision；export 只是一份 final snapshot。
- 不用标题、username、`from_id` 或作者字段推导频道身份。

## 3. 约束与既有合同（Constraints）

- 历史回填已确认采用 Telegram Desktop JSON；Bot 只负责后续持续归档。
- Desktop 与 Bot 必须共用规范化/写入管线，以频道 ID + 消息 ID 幂等合并。
- provenance 至少区分 `telegram_bot_update` 与 `telegram_desktop_json`。
- revision 不可变，current projection 独立；删除/tombstone 属于 G2.2。
- allowlist default-deny；disabled 只停止未来 live ingestion，不冻结已归档内容。
- PostgreSQL 18、Drizzle generated SQL、Node 22、单 Bot/worker 合同保持不变。
- `raw_json` / source evidence 只允许 owner 主动读取，响应 `private, no-store`。
- M1/M2 媒体永久记录首先是元数据与原帖链接；G2.1 不扩大缓存合同。

## 4. 方案设计（Detailed Design）

### 4.1 CLI 合同

```text
kodama import telegram-desktop \
  --input /path/to/result.json \
  --channel -1001234567890 \
  [--channel -1009876543210] \
  [--apply] \
  [--json]
```

- `--input` 必填，只接受普通文件；stdin、目录和 HTML export 不在 G2.1。
- `--channel` 可重复，必须是已存在的 `telegram_channel_allowlist` 公开频道。
- 默认 dry-run；`--apply` 是唯一写入开关。
- 默认输出稳定的人类摘要；`--json` 输出 versioned report JSON。
- dry-run exit 0 表示可干净应用，exit 2 表示存在 conflict/item error，exit 1 表示 fatal。
- apply clean/replay exit 0，完成但有 conflict/item error exit 2，fatal/interrupted exit 1。

dry-run 不创建 run、audit 或临时数据库状态。apply 必须重新解析并重新比较数据库，不把之前一次
dry-run 视为授权或快照。

### 4.2 频道发现与身份映射

parser 第一遍只发现 chat envelope：

```ts
interface DesktopChatDescriptor {
  source: 'root' | 'chats' | 'left_chats';
  index: number | null;
  id: bigint;
  type: string;
  name: string;
}
```

当前 `public_channel` bare ID `1234567890` 的 canonical Bot ID 为
`-(1_000_000_000_000n + 1234567890n)`，显示形式为 `-1001234567890`。实现必须使用 bigint/string，
不能经过 JS number。

每个 `--channel` 必须同时满足：

1. 在 allowlist 中存在；
2. export 中恰好有一个 `public_channel` descriptor 映射到它；
3. 没有重复或不一致 source chat。

private channel/group/DM 不能通过显式参数绕过。导入可创建缺失的 `telegram_channels` archive row，
但使用 allowlist 的 title/username 作为权威元数据；Desktop 不得用 null 覆盖数据库 username。

### 4.3 流式解析

采用 `@streamparser/json-node` 的 path selector 和 Node stream backpressure，不使用整文件
`readFile + JSON.parse`。

解析分两类 pass：

1. descriptor pass：只选择根 chat 字段及 `chats.list.*` / `left_chats.list.*` 的 id/type/name，
   不 materialize messages；
2. message pass：根据已确认 descriptor index 动态选择 `$.messages.*` 或
   `$.chats.list.<index>.messages.*` / `$.left_chats.list.<index>.messages.*`。

每次只产生一条完整 message record。错误样例、conflict 样例和 warning 样例各有固定上限，
report 总体只保存计数与少量无正文标识。测试生成至少 100k messages，验证 RSS/heap 增量有上界。

### 4.4 输入验证与兼容

使用宽松 envelope + 严格 eligible record：

- 允许未知 root/chat/message 字段；
- `id`、`date_unixtime`、`edited_unixtime` 先按十进制字符串读取；
- epoch 字段优先；旧 export 缺少 epoch 时可解析带时区的 ISO `date` / `edited` 并 warning；
- 无时区或非法时间是 item error，不按机器本地时区猜测；
- `type=message` 进入规范化；
- `service`、`unsupported`、`rich_message` 计数跳过；
- 缺 id、越界 id、无正文且无支持媒体、非法结构是 item error；
- 未知 entity/media 保留可见正文与有限 raw provenance，并 warning。

### 4.5 source-neutral domain

现有类型拆成内容快照与来源：

```ts
interface NormalizedMessageSnapshot {
  channel: NormalizedChannelIdentity;
  message: NormalizedMessage;
  media: NormalizedMedia[];
}

type SourceObservation =
  | {
      kind: 'telegram_bot_update';
      sourceKey: string;
      telegramUpdateId: bigint;
      updateType: 'channel_post' | 'edited_channel_post';
      raw: Update;
    }
  | {
      kind: 'telegram_desktop_json';
      sourceKey: string;
      importRunId: string | null;
      sourceChatId: bigint;
      sourceMessageId: bigint;
      raw: DesktopMessageRecord;
    };
```

Bot adapter 与 Desktop adapter 只负责 source-specific parsing。renderer、canonical identity、
revision/media persistence 和 fingerprint 只存在一套实现。

Bot 的既有特殊合同保持：

- 每个新 Bot edit update 都保留 revision，即使 normalized fingerprint 与 current 相同；
- Bot update replay 以真实 update ID/source observation 幂等；
- first-known edit 可成为 revision 1。

Desktop 使用时间/内容决策，不按到达顺序决定 current。

### 4.6 Desktop 文本和 entity

`text_entities` 是权威输入；缺少时回退 `text` string/segment array。

逐 segment 拼接 visible text，并用 JS string `.length` 累计 UTF-16 code unit offset/length，
映射到现有 renderer entity：

| Desktop | Canonical |
| --- | --- |
| `link` | `url` |
| `text_link.href` | `text_link.url` |
| `phone` | `phone_number` |
| `blockquote` + `collapsed=true` | `expandable_blockquote` |
| `custom_emoji.document_id` | `customEmojiId` |
| `pre.language` | `language` |

未知 entity 保留其 text 并 warning，不能删除可见正文。`author` 映射频道签名；
`from` / `from_id` 不作为 author signature。reply/forward 保存 source metadata，但 G2.1 不新增公开
reply/forward API。

### 4.7 fingerprint 与 revision 决策

fingerprint 使用 versioned canonical JSON 的 SHA-256，至少覆盖：

- canonical channel/message ID；
- content kind、text、entities；
- author signature、editedAt；
- source-neutral media metadata；
- 不覆盖 export path、run ID、raw field order、title/username。

锁定 canonical message 后执行：

| 当前状态 | Desktop snapshot | 结果 |
| --- | --- | --- |
| message 不存在 | valid | 创建 message + revision 1 |
| 任意 revision fingerprint 相同 | 任意 | observation 关联已有 revision，`matched` |
| fingerprint 不同 | `editedAt` 严格晚于 current logical time | 新 revision，切 current |
| fingerprint 不同 | `editedAt` 明确更早 | observation 记 `stale`，current 不变 |
| fingerprint 不同 | 时间相同、缺失或不可比较 | observation 记 `conflict`，current 不变 |

既有 message 的 UUID、`publishedAt` 永不被 import 改写。live/import 同时首次创建时，writer 先
`INSERT ... ON CONFLICT DO NOTHING`，再按 `(channel_id, telegram_message_id)` `FOR UPDATE`，
在锁内完成 observation dedupe 和 revision/current 决策。

### 4.8 provenance schema

新增 `import_runs`：

```text
id uuid primary key
source_kind = telegram_desktop_json
source_file_sha256 text
parser_version integer
status running|completed|partial|interrupted
selected_channels jsonb
report jsonb
started_at / completed_at / updated_at
```

新增 `message_source_observations`：

```text
id uuid primary key
source_kind telegram_bot_update|telegram_desktop_json
source_key text
channel_id / message_id
revision_id nullable
import_run_id nullable
telegram_update_id nullable
telegram_message_id bigint
content_fingerprint text
resolution created|matched|stale|conflict
observed_at timestamptz nullable
source_metadata jsonb
raw_json jsonb
created_at
unique(source_kind, source_key)
```

Desktop `sourceKey` 包含 canonical channel ID、message ID、logical edit time 与 fingerprint；
相同 export observation 重放 no-op，不同 final snapshot 能成为新 evidence。

`message_revisions.telegram_update_id` 改为 nullable，保留 FK 与 existing API compatibility；
migration 为所有既有 revision 回填 `telegram_bot_update` observation。

`message_media`：

- `telegram_file_id` / `telegram_file_unique_id` 改 nullable；
- 新增 source kind、private relative path、source media type、availability reason 与有限 metadata；
- check constraint 保证 Bot media 仍有真实 file IDs，Desktop media 不伪造；
- public API 不返回路径/raw；Admin provenance 继续主动加载且 `private, no-store`。

observation 的 `source_metadata` 只保存有界、规范化的来源字段。Desktop 当前保存
`forwardedFrom` 与十进制字符串形式的 `replyToMessageId`；Bot 使用空对象。完整 source record
仍只保存在 owner-only `raw_json`。

### 4.9 媒体路径

G2.1 不访问媒体二进制。记录 source path 前必须：

- 拒绝绝对路径、空路径、NUL、`..` segment；
- 规范化为 export-root relative POSIX path；
- 不把 export root 或本机绝对路径写入 DB/report/log；
- path 仅作为 G2.3 的 untrusted reference，G2.3 实际读取时还必须 realpath containment 并拒绝
  symlink escape。

Telegram 的 not included / exceeds maximum size / unavailable 占位值映射为明确 availability reason。
export 无 `media_group_id`，必须写 null；不得按时间邻近猜相册。

### 4.10 批次、可续跑与锁

只使用独立 import advisory lock，防止两个 import apply 同时运行；不获取 worker/poller leader lock。
dry-run 不获取写锁。

apply 流程：

1. descriptor + validation pass，fatal identity/file/JSON 错误发生在任何业务写入前；
2. 创建 `import_run(status=running)`；
3. 第二次流式 message pass，以有限 batch transaction 写入，每条使用 savepoint 隔离 item error；
4. 每批提交后更新 run counts/checkpoint；
5. 完成后写 `completed` 或 `partial` report。

数据库/进程中断可能留下已提交 batch，但 source observation 唯一约束保证同输入可安全重跑。
importer 会在每批提交前后及最终状态更新前校验独立 session lock。session lock 不能抢占已经发往
另一数据库连接的语句，因此锁若在批次执行中丢失，该批次仍可能提交；提交后校验会终止流程并尝试把
run 标记为 `interrupted`，数据库本身不可用时则保留上一个 `running` checkpoint。这就是 G2.1 的
practical fencing boundary，重跑仍由 source observation 唯一约束收敛。
G2.1 不承诺整文件原子性；这是为了支持官方面向百万消息的 export，避免超长事务和内存放大。

### 4.11 report 与隐私

versioned report 至少包含：

```text
schemaVersion, mode, runId?, fileSha256, parserVersion
selected/source chats
scanned, eligible, createdMessages, createdRevisions, matchedExisting
stale, conflicts, skippedService, skippedUnsupported, warnings, itemErrors
mediaMetadata
bounded issues [{sourceChatId, sourceMessageId?, code, severity, sanitizedReason}]
status, startedAt, completedAt
```

禁止进入 report/log：

- message text/entity raw、完整 message record；
- account contacts、session、未选择 chat；
- secrets、绝对路径、媒体内容；
- SQL/driver 原始错误中可能出现的连接信息。

apply raw evidence 只保存被选择公开频道的 eligible message observation；不存完整 export root。

## 5. 备选方案与权衡（Alternatives Considered）

| 方案 | 优点 | 风险/代价 | 结论 |
| --- | --- | --- | --- |
| 伪造负 `update_id` / file ID | 改动少 | 污染 Bot namespace，provenance 虚假 | 不采用 |
| 独立 import message 表 | live 不动 | 两套 renderer/API/identity，后续难合并 | 不采用 |
| source-neutral snapshot + observation | 一套内容模型，来源真实 | 需要迁移/backfill | 采用 |
| 整文件 `JSON.parse` + 单事务 | 原子、实现简单 | 百万消息内存/长事务不可接受 | 不采用 |
| 流式 bounded batch | 常量内存、可续跑 | apply 可 partial，需 report/idempotency | 采用 |
| 自动导入所有 public channels | 少参数 | 全账号 export 隐私和误导入风险 | 不采用 |
| username/title 身份匹配 | 直观 | 可变、缺失、冲突 | 不采用 |
| ambiguous snapshot 覆盖 current | “最新”看似完整 | 无时间证据，可能回退 live truth | 不采用 |

## 6. 横切关注点（Cross-cutting Concerns）

- **安全/隐私**：只处理显式频道；root PII 不落库；raw owner-only；路径不可信。
- **一致性**：canonical message row lock；observation unique；current 由 logical time 决定。
- **并发**：import 与 live 可并行；import 不动 Bot cursor/inbox/heartbeat。
- **性能**：流式 path selector、有限 batch/savepoint、有限 report samples。
- **可观测性**：持久化 apply run/report；exit 0/1/2 可供脚本判断。
- **演进**：G2.2 消费 stale/conflict evidence 对账；G2.3 消费 private media reference。

## 7. 上线与回滚（Rollout & Migration）

上线：

1. 备份 PostgreSQL；
2. 运行 additive migration 与 Bot revision provenance backfill；
3. 运行 fixture dry-run；
4. 对真实 export 先默认 dry-run，审阅 selected chats 与 report；
5. 显式 `--apply`；
6. 重跑 dry-run，确认 created 收敛为 matched。

回滚：

- 回退应用版本时保留 additive import/provenance tables；
- `message_media.source_kind` 的数据库默认值保持旧应用 Bot media 写入兼容；
- 每次当前应用启动迁移时会幂等修复回滚期间旧应用写入但缺少 observation 的 Bot revision；
- 旧应用仍读取现有 Bot revisions；nullable update ID 不影响已回填 Bot 行；
- 不自动删除已导入消息/revision；
- 若必须撤回数据，以 run provenance 另行设计受审计的反向操作，不在 G2.1 提供 purge。

## 8. 测试策略（Testing）

### Unit/fixture

- 单频道与 `chats.list` / `left_chats.list`；
- text string/array/segments、emoji UTF-16、link/pre/blockquote/custom emoji；
- edited、reply、forward、author；
- photo/document/animation/audio/video/voice/sticker/video message 与三种 placeholder；
- service、unsupported、rich message、未知 entity/media；
- 旧 export 的 ISO fallback、坏日期、越界 ID、重复 ID；
- absolute/`..`/NUL/symlink-style path；
- 生成 100k+ messages，断言 bounded memory 与 bounded issue samples。

### PostgreSQL 18 integration

- migration/backfill、重复 migration；
- dry-run 前后相关表零变化；
- apply 后同文件与 overlapping export replay；
- export→live、live→export、新旧 edit、同时间 conflict；
- live/import concurrent first insert；
- batch 中断后重跑；
- imported current 的 Admin provenance/raw；
- allowlist disabled 可显式导入，但 allowlist/cursor/inbox/heartbeat 均不变；
- 既有 message UUID/publishedAt 与公开 pagination 稳定。
